# Модель данных инструмента управления политиками и настройками KSC for Linux

## 0. Назначение и принципы

Инструмент заменяет Excel-матрицу и бумажные регламенты полноценной системой управления АВПО: политики, задачи, автоматизации, change management и документация в одном месте, с наглядным UI вместо табличной простыни KSC.

**Источник правды.** Инструмент хранит только *конфигурацию* (что должно быть настроено), а не фактическое состояние устройств — это остаётся в самом KSC. Инструмент не подключается к устройствам и не синхронизирует реальные теги/статусы; он про проектирование, документирование и согласование настроек.

**Принцип "каталоги — это данные, а не код".** Всё, что может измениться с выходом новой версии KESL/Агента/KSC (новый параметр, новое событие, новая кнопка, новый тип критерия автоматизации), должно добавляться как строка в справочник, а не как новый код или миграция схемы. UI рендерит формы по метаданным (`value_type`, `options`, `unit`), а не по хардкоду конкретных полей. Каждая запись справочника несёт версии `added_in` / `deprecated_in` / `removed_in`, поэтому старые политики не ломаются при обновлении каталога, а новые параметры появляются сразу во всех местах, где они нужны (форма, матрица сравнения, подсказки).

## 2. Слой "Автоматизации" — единый движок правил

Вместо отдельной таблицы под каждый вид автоматизации — общая модель "условие → действие", т.к. moving rules, tagging rules, task triggers и device selections в KSC имеют одинаковую анатомию.

| Сущность | Ключевые поля | Комментарий |
|---|---|---|
| `AutomationRule` | id, environment_id, kind (`moving_rule` \| `tagging_rule` \| `task_trigger` \| `device_selection` \| `discovery_poll`), name, priority, enabled, condition (json, построено из `CriteriaCatalog`), action (json: move_to_group / assign_tag / run_task / notify) | Универсальная запись правила. |
| `CriteriaCatalog` | id, criteria_key, group (сеть / AD / приложение / защита / виртуализация), value_type, operator_options[], description | Справочник условий отбора устройств. Пополняется по документации KSC (AD OU/группа, IP/DNS/домен, статус компонента приложения, статус защиты, тип ВМ и т.д.) без изменения кода конструктора условий. |
| `TaskTriggerCatalog` | id, trigger_type (`schedule` \| `on_event` \| `on_task_complete` \| `on_app_launch` \| `after_db_update`), params_schema | Справочник видов запуска задач (вирус-вспышка, завершение другой задачи, запуск приложения, обновление баз и т.д.), используется полем `action` в `AutomationRule` с kind=`task_trigger`. |
| `DiscoveryPollConfig` | id, environment_id, source_type (`ad_domain` \| `ip_range` \| `network_neighborhood`), schedule, target_group_id | Опрос сети/AD для обнаружения устройств — питает moving rules. |

## 4. Слой Change Management — полноценный workflow

| Сущность | Ключевые поля | Комментарий |
|---|---|---|
| `ChangeRequest` | id, title, description, author_id, created_at, status (`draft`→`in_review`→`approved`→`applied`, либо `rejected`/`rolled_back`), related_ticket | Может затрагивать несколько сущностей сразу. |

## 6. Как это защищает от "развала" при обновлениях KSC

- Новый параметр политики → строка в `ParameterCatalog` → сразу доступен в форме, матрице сравнения и подсказках, без правки кода.
- Новое событие → строка в `EventCatalog`.
- Новый критерий автоматизации → строка в `CriteriaCatalog` → сразу доступен в конструкторе условий `AutomationRule`.
- Новый вид триггера задачи → строка в `TaskTriggerCatalog`.
- Версии (`added_in`/`deprecated_in`/`removed_in`) не дают новым записям ломать существующие политики, привязанные к более старой версии продукта.

## 7. Матрица ролей KSC (RBAC самого KSC — не путать с ролями Change Management из раздела 4)

Важное разделение: роли `editor/reviewer/approver/viewer` из раздела 4 — это роли **внутри нашего инструмента** (кто может согласовывать изменения конфигурации). Раздел ниже — про **встроенную ролевую модель самого KSC**: кто из живых пользователей KSC что может делать в консоли. Наш инструмент должен уметь документировать и настраивать именно эту матрицу, потому что в консоли KSC она размазана по вкладке "Права" с раскрывающимися деревьями — без документации не разобраться, какая галка что открывает.

### 7.1 Базовые права (применяются почти везде)

| Право | Что дает | Что блокирует, если не выдано |
|---|---|---|
| **Read** | Просматривать настройки объектов. Обязательно для самого подключения к Administration Server. | Без Read пользователь не подключится вообще. |
| **Write** | Менять настройки, создавать новые объекты. | Без Write — только смотреть, ничего не создать и не изменить. |
| **Execute** | Запускать операции (задачи, отчёты) на объектах. | Без Execute — видит задачу/отчёт, но не может запустить. |
| **Perform operations on device selections** | Доп. право, обязательное вместе с Read/Write/Execute, если задача/отчёт/настройка применяется не к статичной группе, а к динамической выборке устройств (device selection). | Без него — задачи и отчёты, привязанные к выборкам, недоступны, даже если Execute есть. |

Если ни Allow, ни Deny не выбраны явно — право считается **запрещённым** (deny by default). Это частая причина "у пользователя все галки стоят, а функция не работает" — где-то по дереву прав унаследован Deny.

### 7.2 Функциональные области (Functional areas) — что именно включает каждая

**General features (общие для всех редакций):**

| Область | Что дает пользователю | Типичный сценарий, когда нужна |
|---|---|---|
| Basic functionality | Базовые операции: загрузка обновлений в репозиторий, доставка отчётов, стандартные отчёты о защите/угрозах/событиях. Права здесь наследуются почти всеми задачами/отчётами, которые не выделены в отдельную область. | Нужна почти любому пользователю KSC. |
| Management of administration groups | Добавление/удаление устройств и подгрупп в дереве групп. | Администратор, который переструктурирует дерево групп. |
| Access objects regardless of their ACLs | Доступ ко всем объектам вне зависимости от их персональных списков доступа (ACL). | Аудитор, который должен видеть всё, даже то, что закрыто от других. |
| Deleted objects | Просмотр/очистка Корзины удалённых объектов. | Редко нужна кому-то кроме главного администратора. |
| Event processing | Менять настройки регистрации событий, уведомлений, удалять события, настраивать пороги "вирусной вспышки". | Тот, кто настраивает алертинг и SIEM-интеграции. |
| Operations on Administration Server | Бэкап/обслуживание БД сервера, настройка портов подключения. | Инфраструктурный администратор KSC, не связанный с антивирусной политикой. |
| Kaspersky software deployment | Развёртывание приложений Kaspersky, управление патчами. | Тот, кто занимается инсталляциями/обновлениями агентов. |
| License key management | Экспорт файла ключа, изменение настроек лицензии. | Обычно только главный администратор — щедро не раздавайте. |
| Enforced report management | Создавать/выполнять отчёты вне зависимости от их ACL. | Аудитор/руководитель, которому нужны все отчёты без ограничений. |
| Hierarchy of Administration Servers | Регистрация/удаление вторичных серверов. | Актуально только при иерархии из нескольких серверов. |
| User permissions | Управление ролями, локальными пользователями KSC, группами безопасности — то есть право **раздавать права другим**. | Только доверенный администратор безопасности. |
| Virtual Administration Servers | Управление виртуальными серверами (для мультитенантности). | Актуально, если у вас несколько виртуальных серверов под разных заказчиков. |
| Encryption Key Management | Экспорт/импорт ключей шифрования. | Актуально для Windows-агентов с BitLocker; для чистого Linux почти не используется. |

**System management (актуально для Linux):**

| Область | Что дает | Комментарий по Linux |
|---|---|---|
| Connectivity | RDP-сессии, туннелирование, сохранение файлов с устройства на рабочую станцию администратора. | RDP/туннелирование — фактически для Windows-устройств; на Linux ограниченно применимо. |
| Hardware inventory | Просмотр/изменение реестра оборудования, связанные отчёты. | Работает и для Linux-агентов. |
| Software inventory | Инвентаризация ПО, отчёты об установленных приложениях. | Работает и для Linux-агентов. |
| Operating system deployment, Vulnerability and patch management, Remote installation, Network Access Control | В основном ориентированы на Windows-инфраструктуру (PXE, Windows Update, NAC через Cisco). | Для чистого Linux-парка обычно не нужны — не тратьте время на их настройку, если у вас не гибридная сеть. |

**Mobile Device Management** — в KSC Linux эта функциональная область недоступна вовсе (подтверждено документацией по KSC 13 Linux), выдавать по ней права не нужно.

### 7.3 Предустановленные роли KSC — что выбрать без чтения документации

| Роль | Кому подходит | Что разрешает |
|---|---|---|
| **Administration Server Administrator** | Технический администратор инфраструктуры KSC (не антивирусной политики) | Полный доступ к Basic functionality, обработке событий, иерархии серверов, виртуальным серверам, System management: Connectivity/Hardware/Software inventory; чтение+запись по ключам шифрования. |
| **Kaspersky Endpoint Security Administrator** | Тот, кто ведёт именно антивирусные политики (это, скорее всего, ваша роль) | Полный доступ ко всем функциям KESL/области защиты + Basic functionality. |
| **Kaspersky Endpoint Security Operator** | Дежурный оператор, который работает с готовыми политиками, но не создаёт новые | Read+Execute на функциях KESL — может запускать задачи и смотреть отчёты, не может менять политики. |
| **Auditor** | ИБ-аудитор / служба внутреннего контроля | Видит и выполняет всё, включая объекты, закрытые ACL от других; специально предупреждение — назначать ограниченному кругу. |
| **Supervisor** | Руководитель ИБ без операционных задач | Только просмотр (Read), включая объекты вне ACL и отчёты. Ничего не меняет и не запускает. |
| **Security Officer** | Офицер ИБ, отвечающий за отчётность и мониторинг | Все операции просмотра + управление отчётами + права на Connectivity (RDP/туннели/сохранение файлов) для расследований инцидентов. |
| **Main Administrator** | Главный администратор организации | Всё, кроме "Access objects regardless of ACL" и "Enforced report management" — то есть уважает персональные ограничения доступа других пользователей. |
| **Main Operator** | Дежурный на всём стеке (антивирус + инфраструктура) | Read+Execute почти везде, без права создавать/менять. |
| **Installation Administrator / Operator** | Тот, кто занимается развёртыванием агентов и патчей | Права на Kaspersky software deployment, remote installation; Operator — без права утверждать патчи. |

## 8. Опросы сети и домена (Discovery) — специфика Linux-редакции

Важно: в **KSC Linux** набор методов обнаружения устройств меньше, чем в Windows-версии — NetBIOS-опрос "Windows-сети" (quick/full poll) ориентирован на Windows Administration Server и там не является основным механизмом. Для Linux-редакции реально используются три метода:

| Метод | Что делает | Когда включать | Ключевые настройки |
|---|---|---|---|
| **Опрос контроллера домена** (Domain controller polling — AD или Samba 4) | Получает структуру доменных OU, учётные записи, группы безопасности и DNS-имена устройств, входящих в домен. | Включайте, если весь парк устройств состоит в домене (AD или Samba). Это основной способ узнать "что у нас есть" в доменной инфраструктуре. | Адрес и учётные данные контроллера домена (для Linux distribution point — только явно указанные домены, без опроса "всего леса"); расписание опроса (по умолчанию раз в час); опрос можно выполнять и через distribution point. |
| **Опрос IP-диапазонов** (IP range polling) | Посылает ICMP-запросы (ping) по указанным диапазонам IP, дополнительно делает обратное DNS-резолвинг перед пингом. | Включайте, если есть устройства **вне домена**, либо для перекрёстной проверки состава сети. Не рекомендуется как единственный метод, если у вас уже есть опрос домена — данные будут дублироваться. | Список диапазонов (по умолчанию — тот, где стоит сам сервер); расписание (по умолчанию каждые 420 минут); требует корректно настроенной **обратной зоны DNS** — без неё метод не найдёт ничего. |
| **Zeroconf-опрос** (только для Linux distribution point, IPv6) | Обнаруживает устройства в IPv6-сети через zero-configuration networking, без ручной настройки диапазонов. | Включайте, только если в сети реально используется IPv6 и хотите его покрыть; недоступно, если distribution point не на Linux. | Включение/выключение переключателем; своих доп. настроек по диапазонам не требует — сеть определяется автоматически. |

**Что происходит с найденными устройствами:** если настроены и включены правила автоматического перемещения (см. раздел 2, `AutomationRule` kind=`moving_rule`), новые устройства попадают сразу в целевую группу; если правил нет — они падают в группу "Нераспределённые устройства" и требуют ручной обработки.

### Модель данных

| Сущность | Ключевые поля |
|---|---|
| `DiscoveryPollConfig` | id, environment_id, poll_type (`domain_controller`\|`ip_range`\|`zeroconf`), enabled, schedule (json), executor (`admin_server`\|`distribution_point`, ref), description_note |
| `DomainControllerTarget` | poll_config_id, domain_or_controller_address, auth_ref, directory_kind (`active_directory`\|`samba`) | Для `domain_controller`, поддерживает несколько контроллеров на один environment. |
| `IpRangeTarget` | poll_config_id, ip_range_start, ip_range_end (или CIDR), reverse_dns_required (bool, для памятки) | Для `ip_range`. |

Как и везде выше — если в новой версии KSC появится четвёртый метод опроса, это новая строка в справочнике `poll_type` (расширяемый enum/каталог), а не новая таблица и не новый код формы.

## 9. Мониторинг и отчётность (Дашборд, виджеты, отчёты)

### 9.1 Дашборд и виджеты

Дашборд KSC — набор настраиваемых виджетов (круговые/кольцевые диаграммы, таблицы, графики, столбчатые диаграммы, списки), данные обновляются автоматически раз в 1-2 минуты. У каждого виджета отдельно настраивается тип отображения (например, для показателя можно переключить между "столбцы" и "линии") и фильтры (период, правило подсчёта объектов).

**Режим "только дашборд"** — специальный режим для людей, которым не нужно управлять сетью, но нужно видеть статистику (типичный пример — руководитель): пользователю показывается только предустановленный набор виджетов без главного меню, он не может их добавлять/скрывать/двигать. Настраивает это администратор заранее.

Это прямо стыкуется с ролями из раздела 7: пользователь в роли `Supervisor` — кандидат на "только дашборд".

### 9.2 Отчёты

Отчёт в KSC = шаблон (report template) + при желании задача доставки (report delivery task, отправка по расписанию, например на почту). Жизненный цикл: создание шаблона → просмотр/правка свойств → генерация → экспорт в файл (PDF/HTML/XLS) → опционально регулярная доставка.

### 9.3 Модель данных

| Сущность | Ключевые поля | Комментарий |
|---|---|---|
| `DashboardLayoutDefinition` | id, assigned_role_or_user_ref, widgets[] (widget_type_ref, settings json, position) | Документирует, что видит конкретная роль/пользователь на дашборде — особенно важно для "только дашборд". |
| `WidgetTypeCatalog` | id, name, display_kind (`pie`\|`donut`\|`table`\|`graph`\|`bar`\|`list`), description | Каталог типов виджетов — новый тип виджета в новой версии = новая строка. |
| `ReportTemplateDefinition` | id, name, report_type_ref, scope (группа/выборка устройств), filters (json), export_formats[] | Что должно быть настроено как шаблон отчёта. |
| `ReportTypeCatalog` | id, name, category, description | Каталог видов отчётов (по защите, по угрозам, по инвентаризации и т.д.), пополняемый. |
| `ReportDeliveryTask` | id, report_template_id, schedule (json), recipients[], format | Регулярная рассылка отчёта. |

## 10. Backup и отказоустойчивость Administration Server

Резервное копирование выполняется через задачу бэкапа или утилиту `klbackup`. В копию попадает: база данных Administration Server (политики, задачи, настройки приложений, события, хранящиеся на сервере), конфигурация структуры групп управления и устройств, репозиторий пакетов удалённой установки, сертификат сервера. **Не попадают** установленные плагины управления — после восстановления их нужно скачать и переустановить заново.

Важные практические моменты, которые стоит зафиксировать как регламент, а не только как техническую настройку:
- По умолчанию задача бэкапа создаётся **с пустым паролем** — без явной установки пароля все сертификаты и ключи лицензий в копии хранятся незашифрованными. Это прямая находка для регламента ИБ: пароль обязателен.
- Резервную копию рекомендуется делать не только по расписанию, но и **перед каждым значимым изменением** — обновлением сервера, установкой патча. Это естественно стыкуется с нашим Change Management (раздел 4): для заявок с высоким уровнем риска можно требовать отметку "бэкап сделан" перед статусом "применено".
- Восстановление возможно только на Administration Server той же или более новой версии и с той же СУБД (MySQL/MariaDB) того же или более позднего релиза — при миграции на другое железо это нужно проверять заранее.

### Модель данных

| Сущность | Ключевые поля | Комментарий |
|---|---|---|
| `BackupPolicyDefinition` | id, environment_id, schedule (json), storage_target (`local_path`\|`s3`\|`azure`), password_protected (bool, должен быть true), retention_days | Регламент бэкапа как конфигурация, которую можно проверить на соответствие политике ИБ. |
| `BackupChecklistItem` (связь с `ChangeRequest`) | change_request_id, backup_confirmed (bool), backup_reference | Необязательное required-поле для заявок с высоким риском — не даёт перевести заявку в "применено" без подтверждённого бэкапа. |

## 11. Иерархия Administration Server (первичный / вторичный / виртуальный)

Иерархия серверов нужна для снижения нагрузки на один сервер, работы с удалёнными офисами по медленным каналам и разделения зон ответственности между администраторами. Три роли сервера:

- **Первичный (primary/master)** — верхний уровень; Linux-редакция первичного сервера может управлять и Linux-, и Windows-вторичными серверами; Windows-первичный может управлять Linux-вторичным.
- **Вторичный (secondary/slave)** — подчинённый сервер, обычно в отдельном офисе/сегменте сети.
- **Виртуальный (virtual)** — частный случай вторичного, создаётся на базе одного физического сервера. У каждого виртуального сервера — **свои** дерево групп, список нераспределённых устройств, отчёты, выборки устройств, события, пакеты установки, правила перемещения. Это готовый механизм изоляции клиентов/сред внутри одного физического KSC — прямое соответствие нашей сущности `Environment` из раздела 1: если у вас в KSC уже используются или планируются виртуальные серверы под разные среды/заказчиков, они логично мапятся 1:1 на `Environment` в нашей модели.

**Важный нюанс наследования:** политики и групповые задачи первичного сервера транслируются на вторичные и отображаются там как заблокированные для изменения — то есть управлять из "дочернего" сервера можно только тем, что не идёт сверху. Это стоит учитывать в модели `Policy.parent_policy_id` — цепочка наследования может идти не только по дереву групп внутри одного сервера, но и по иерархии серверов.

### Модель данных

| Сущность | Ключевые поля | Комментарий |
|---|---|---|
| `AdministrationServerNode` | id, role (`primary`\|`secondary`\|`virtual`), parent_server_id, environment_id (для virtual — прямая связь с `Environment`), connection_address, os_kind (`linux`\|`windows`) | Описывает топологию серверов как конфигурацию (для документации архитектуры, не для реального подключения). |
| `ServerHierarchyPolicyLink` | policy_id, source_server_id, propagated_to_server_id, locked (bool) | Фиксирует, какая политика транслируется вниз по иерархии и заблокирована там для правок. |

## 12. Наследование политик — как это на самом деле работает (три уровня, не два)

То, что на первый взгляд выглядит как "две галочки", на самом деле три независимых механизма, которые накладываются друг на друга. Путаница в реальном KSC возникает именно потому, что они физически расположены в разных местах интерфейса.

### 12.1 Уровень 1 — замочек на конкретном параметре

У каждого параметра политики есть атрибут "заблокировано для изменения в дочерних политиках и в локальных настройках приложения" (замочек). Это свойство самого параметра, задаётся в родительской политике point-by-point. Замочек **не имеет смысла сам по себе** — он активируется только если для дочерней политики включено наследование (уровень 2).

### 12.2 Уровень 2 — два переключателя политики

- **"Наследовать настройки от родительской политики"** (Inherit settings from parent policy) — переключатель на самой дочерней политике. Включён → значения параметров, заблокированных замочком в родительской политике, нельзя поменять в дочерней. Незаблокированные параметры менять можно. Выключен → дочерняя политика полностью свободна, даже если в родителе что-то заблокировано.
- **"Принудительно транслировать настройки дочерним политикам"** (Force inheritance of settings in child policies) — переключатель на **родительской** политике. Включение автоматически включает "Наследовать" у всех дочерних политик и **блокирует им возможность его отключить**. Это способ администратора наверху гарантировать, что вниз по иерархии никто не отвяжется от заблокированных настроек.

Практическое следствие: если нужно железно продавить настройку по всей организации — не полагайтесь на замочек в одиночку, включайте и "Принудительно" на родителе, иначе локальный администратор subgroup просто отключит у себя наследование и обойдёт замочек.

### 12.3 Уровень 3 — иерархия групп и серверов как таковая

"Родительская" и "дочерняя" политика — это не абстракция, а политики, физически созданные в родительской и дочерней **группе управления** (см. раздел 1, `AdministrationGroup`) либо на родительском/дочернем **сервере** в иерархии (раздел 11). Если в дочерней группе для приложения вообще нет своей политики — устройства этой группы получают настройки политики из ближайшей группы-предка, где такая политика есть и активна. Здесь работает отдельное правило: **на одной группе может быть активна только одна политика на одно приложение одновременно** — у политики есть статус `Активна` / `Неактивна` (заготовка на экстренный случай, например заранее настроенная политика "заблокировать все флешки" на случай вспышки) / `Не в офисе` (включается, когда устройство выходит за периметр сети; не участвует в наследовании по иерархии).

### 12.4 Модель данных

| Сущность | Ключевые поля | Комментарий |
|---|---|---|
| `PolicySetting` (уточнение к разделу 3) | + `locked` (замочек на параметре в этой политике) | Замочек — атрибут конкретного значения параметра в конкретной политике, а не глобальная настройка. |
| `Policy` (уточнение к разделу 3) | + `status` (`active`\|`inactive`\|`out_of_office`), `inherit_from_parent` (bool, переключатель уровня 2а), `force_inherit_to_children` (bool, переключатель уровня 2б), `parent_policy_id` (вычисляется из `target_group_id`/иерархии, не задаётся вручную) | `parent_policy_id` — не свободное поле: правильная политика-родитель однозначно определяется положением группы/сервера в дереве, инструмент должен это вычислять, а не давать выбрать вручную (иначе разойдётся с реальным KSC). |
| Вычисляемая функция `resolve_effective_settings(policy_id)` | — | Не сущность, а обязательная бизнес-логика: для каждого параметра пройти вверх по цепочке (группа → родительская группа → сервер → родительский сервер), найти первого предка с активной политикой, применить его значение, если оно заблокировано и наследование включено; иначе взять значение из самой политики. Это то, что должно выводиться в UI как "эффективное значение" — то, что реально долетит до устройства. |

## 13. Размещение политик по группам администрирования

Правило простое, но именно оно определяет, сколько политик реально понадобится: **политика создаётся в конкретной группе и действует на все устройства в этой группе и во всех её подгруппах**, если в подгруппе нет собственной активной политики того же приложения (тогда работает правило из 12.3 — заменяет её). Отсюда вытекает практический принцип проектирования дерева групп из вашей таблицы (раздел 1): группа нужна не просто для организационного удобства, а именно там, где требуется **другой набор настроек защиты**. Если у двух групп одинаковая политика — это сигнал, что дробление избыточно и можно объединить их под одной политикой уровня выше.

В модели данных это уже отражено — `Policy.target_group_id` и есть точка привязки; отдельной сущности не требуется, важно только, чтобы UI визуализировал дерево групп с политикой, реально действующей на каждую (включая унаследованную сверху), а не только явно созданные в этой группе.

## 14. KSN-интеграция

Два независимых переключателя плюс отдельная архитектура доставки — тоже частый источник путаницы.

### 14.1 Что переключается

| Переключатель | Что делает | На что влияет |
|---|---|---|
| **Использовать KSN** (участие вкл/выкл) | Базовое согласие на обмен репутационными данными с облаком. Без этого — не работает быстрая проверка репутации файлов/сайтов, выше вероятность false positive, медленнее реакция на новые угрозы. | Если выключено на уровне Administration Server — с сервера и управляемых устройств через KSC вообще ничего не уходит; но конкретное устройство может слать напрямую в облако мимо KSC, если так настроено локально в самом приложении — это отдельная настройка внутри политики KESL. |
| **Расширенный режим KSN (Extended KSN mode)** | Помимо базовой репутации, приложение автоматически отправляет расширенную статистику работы и может отправлять сами файлы (или их части), похожие на вредоносные, на дополнительный анализ в Kaspersky. | Резко увеличивает объём и чувствительность передаваемых данных — прямой вопрос к вашей регламентной документации по обработке данных (что именно уходит и куда). |

### 14.2 Global KSN vs Private KSN — принципиальный выбор для регламента

- **Global KSN** — данные уходят в облако Kaspersky. Быстрее реагирует на новые угрозы (глобальная база), но данные покидают периметр организации/страны.
- **Private KSN** — свой локальный сервер репутации внутри организации; устройства получают тот же функционал (репутация файлов/сайтов), но **ничего не уходит наружу** ни от KSC, ни от конечных приложений. Явно предназначен для случаев, когда передача данных за периметр запрещена законом или корпоративной политикой ИБ. Требует отдельной лицензии/файла настроек KSN Proxy.

### 14.3 Архитектура доставки (proxy)

Устройство не стучится в облако Kaspersky напрямую — запрос идёт через **KSN Proxy**: обычно это сам Administration Server (порт **13111** по умолчанию, может быть UDP **15111**), либо distribution point, которому явно разрешили быть проксёй. Если проксирующий сервис недоступен, можно разрешить приложению стучаться в KSN cloud напрямую ("Use KSN servers when KSN Proxy is not available") — с точки зрения регламента это отдельная дырка, которую стоит явно проверять: разрешён ли обход прокси.

### 14.4 Модель данных

| Сущность | Ключевые поля |
|---|---|
| `KsnConfig` (часть `PolicySetting`, но выделено отдельно для наглядности) | policy_id, participation_enabled (bool), extended_mode_enabled (bool), provider (`global`\|`private`), allow_direct_fallback (bool, обход proxy) |
| `KsnProxyInfrastructure` | environment_id, proxy_host_ref (`admin_server`\|`distribution_point`), tcp_port (default 13111), udp_enabled (bool), udp_port (default 15111), private_ksn_config_ref | Отдельно от политики — это инфраструктурная настройка сервера/точки распространения, а не настройка конкретной политики. |

## 15. Плюсы/минусы у каждого пункта защиты

Это новое поле в `ParameterCatalog` (раздел 3): `tradeoffs` = `{pros: [...], cons: [...]}`. Наполнить его для всех ~200 параметров — отдельная содержательная работа (её разумно делать поэтапно, начиная с самых спорных настроек — того, что реально хочется выключить "для скорости"), но вот принцип и несколько эталонных примеров, по которым можно калибровать остальные:

| Параметр | Плюсы включения | Минусы включения (= плюсы выключения, наоборот) |
|---|---|---|
| Проверка внутри архивов | Не даёт вредоносному файлу проехать "в чехле" — упаковка в zip/rar — один из базовых приёмов обхода антивируса. | Заметно увеличивает время сканирования и нагрузку на CPU/IO, особенно на файловых серверах с большим количеством архивов. |
| Расширенный режим KSN | Быстрее получаете вердикт по новым, ещё не описанным в базах угрозам; меньше false positive. | Приложение отправляет расширенную статистику и может отправлять сами файлы во внешнее облако — вопрос комплаенса и объёма трафика. |
| Anti-Cryptor | Защищает файлы на файловом сервере от шифрования шифровальщиком, который атакует по сети (SMB/NFS), даже если сам шифровальщик сидит на незащищённой рабочей станции. | Требует, чтобы Anti-Cryptor мог анализировать сетевые запросы к файлам — дополнительная нагрузка на файловый сервер при большом количестве одновременных сессий. |
| Firewall Management (перехват управления сетевым экраном ОС) | Правила МЭ гарантированно не будут случайно/злонамеренно изменены мимо KSC. | Если правила настроены неточно, легитимный сетевой сервис на этом хосте может неожиданно перестать отвечать — нужно тщательное тестирование перед раскаткой на прод. |
| Device Control — блокировка неизвестных USB по умолчанию | Перекрывает главный вектор заражения через съёмные носители и несанкционированный вынос данных. | Требует заранее продуманного списка доверенных устройств — иначе первый же легитимный внешний диск/токен встанет колом, и это заявки в поддержку. |

### Модель данных

| Сущность | Ключевые поля |
|---|---|
| `ParameterCatalog` (уточнение к разделу 3) | + `tradeoffs_pros[]` (текстовые пункты), `tradeoffs_cons[]`, `tradeoffs_reviewed_by`, `tradeoffs_reviewed_at` | Последние два поля — чтобы отличать "мы это проверили и согласовали формулировку" от "заготовка, ещё не вычитана", раз наполнение растянется по времени. |

## 16. Матрица атак — покрытие и пробелы

Идея: каждый параметр (или конкретное его значение — "включено", "жёсткий режим") может закрывать один или несколько векторов атаки. Тогда для любой конкретной политики можно вычислить, что реально закрыто, что закрыто частично, а что не закрыто вообще — и увидеть это в виде наглядной матрицы (пример визуализации выше в диалоге — именно такой отчёт и должен генерироваться автоматически по текущим значениям политики).

Основа таксономии векторов атаки — не обязательно придумывать с нуля: можно взять за скелет тактики/техники **MITRE ATT&CK** (Initial Access, Execution, Persistence, Defense Evasion, Lateral Movement, Exfiltration, Impact и т.д.), адаптировав под то, что реально видно на уровне endpoint-защиты, а не всей кибербезопасности сразу.

### Модель данных

| Сущность | Ключевые поля | Комментарий |
|---|---|---|
| `ThreatVectorCatalog` | id, tactic (макрокатегория, например по MITRE), name, description, mitre_technique_ref (опционально, для трассируемости) | Расширяемый справочник — новый вектор атаки = новая строка, не новый код. |
| `ControlMapping` | id, parameter_id, required_state (json: например `enabled=true` или `value=strict`), threat_vector_ids[], coverage_strength (`prevents`\|`detects`\|`partial`\|`compensating`), notes | Связка "при таком состоянии параметра — вот что закрывается и насколько сильно". Один параметр может закрывать несколько векторов; один вектор обычно закрывается несколькими параметрами (defense in depth). |
| Вычисляемый отчёт `PolicyCoverageReport(policy_id)` | Не хранимая сущность, а расчёт по `resolve_effective_settings` (раздел 12.4) × `ControlMapping`: для каждого `ThreatVectorCatalog` — статус `covered` (есть хотя бы один prevents/detects-контроль в нужном состоянии) / `partial` (есть compensating, но не prevents) / `gap` (ничего не закрывает) | Это и есть тот самый наглядный отчёт "как политика закрывает матрицу атак", который вы просили — таблица/тепловая карта векторов × статус, с раскрытием по клику "какие именно настройки дают это покрытие". |

Наполнение `ControlMapping` — как и с `tradeoffs`, содержательная работа, которую разумно вести поэтапно, начиная с самых частых сценариев атак на Linux-инфраструктуру (компрометация через SSH/сеть, шифровальщики через сетевые диски, несанкционированные съёмные носители, веб-угрозы через прокси/загрузку файлов) — то есть с того, что уже есть в вашей исходной таблице, а не с полного покрытия MITRE ATT&CK сразу.

## 17. Архитектура и стек для MVP

Ограничения: изолированная сеть без интернета в проде, соло-разработка с AI-инструментами (Claude Code), весь описанный охват должен войти в итоговую версию.

### 17.1 Стек

- **TypeScript везде** — от БД до фронтенда один язык, меньше рассинхронизации типов, максимально предсказуемо для AI-ассистентов.
- **Postgres** — реляционные связи + JSONB для гибких полей (`row_data`, `condition`/`action`, `list_schema`) в одной СУБД.
- **Prisma** — schema-first ORM, миграции, TS-типы генерируются из схемы автоматически.
- **tRPC** — типизированный API без отдельного слоя OpenAPI/GraphQL-схем.
- **React + Vite + Tailwind + shadcn/ui** — фронтенд, все зависимости из npm, ничего с CDN в runtime.
- **TanStack Query** (серверный стейт) + **React Hook Form + Zod** (формы; Zod-схема валидации генерируется динамически из `ParameterCatalog`, а не пишется руками под каждый параметр).

### 17.2 Структура репозитория

```
/apps/web        — React SPA
/apps/api        — tRPC-сервер (Node)
/packages/domain — чистая бизнес-логика: resolve_effective_settings,
                   расчёт PolicyCoverageReport, стейт-машина ChangeRequest
/packages/db     — Prisma schema + клиент
/catalogs        — версионируемые YAML/JSON (параметры, события, критерии,
                   векторы атак, роли KSC) — правятся через git PR,
                   сидер синхронизирует их в БД при деплое
/infra           — Dockerfile, docker-compose, скрипты для переноса образа
```

### 17.3 Изоляция от интернета

- Все npm-зависимости фиксируются в lock-файле; в проде `npm install` не выполняется — разворачивается готовый собранный образ.
- Шрифты/иконки вендорятся в билд локально, ни одного внешнего запроса в runtime.
- Auth на MVP — локальный логин/пароль (bcrypt + сессии), без OAuth/SSO. Интеграция с AD — расширение после MVP.
- Перенос: сборка на машине с интернетом (`docker build`) → `docker save` → перенос образа через утверждённый канал (носитель/диод) → `docker-compose up` в изолированном сегменте без обращений во внешнюю сеть.

### 17.4 Ключевые универсальные компоненты фронтенда

- `<SettingField />` — рендер по `value_type` из `ParameterCatalog` (единый компонент для всех параметров, старых и новых).
- `<ConditionBuilder />` — тот же принцип для условий `AutomationRule`, по метаданным `CriteriaCatalog`.
- Дерево групп + редактор политики (вкладки: Настройки / Списки / События / Задачи) + панель "эффективное значение" (результат `resolve_effective_settings`: унаследовано / заблокировано / переопределено).
- Матрица сравнения параметров × политик с подсветкой отличий от родителя.
- Универсальный diff-вьюер для Change Management (работает одинаково для политик, автоматизаций, задач — у всех diff хранится одинаково).
- Отчёт покрытия атак — таблица-heatmap по `PolicyCoverageReport`.

### 17.5 Порядок сборки (весь функционал входит, но не одновременно)

1. Каталоги + сидер `/catalogs` → БД.
2. Дерево групп + `SettingField` + редактор политики + `resolve_effective_settings` (разделы 1, 3, 12, 13).
3. Матрица сравнения + задачи (раздел 3).
4. Автоматизации + `ConditionBuilder` + опросы сети (разделы 2, 8).
5. Change Management (раздел 4).
6. RBAC KSC, KSN, backup/иерархия серверов (разделы 7, 14, 10, 11) — формы над уже готовым движком.
7. Плюсы/минусы и матрица атак (разделы 15, 16) — контентно самое долгое, наполняется параллельно с шагами 2-6.
