# Информационная система управления политиками безопасности на основе Kaspersky Security Center for Linux

Веб-приложение для моделирования политик **Kaspersky Endpoint Security for Linux (KESL)**,
централизованно управляемых через **Kaspersky Security Center for Linux (KSC)**. Система
воспроизводит структуру и навигацию консоли KSC, вычисляет *действующую* политику с учётом
трёхуровневого наследования и **честно** оценивает, какие атаки политикой реально закрыты.

> Учебный проект. Работает на реальных данных, выгруженных из KSC, и сверен с официальной
> документацией KESL 12.4.

---

## Что решает

Политика KESL — это десятки компонентов и сотни параметров, а их итоговые значения
складываются по наследованию сверху вниз по дереву групп с блокировкой «замочком». В штатной
консоли администратор не видит результирующую политику после наследования и не может оценить,
какие угрозы она реально закрывает. Система устраняет оба пробела.

## Возможности

- **Группы администрирования** — дерево групп KSC внутри среды, с назначенными политиками.
- **Редактор политик** с навигацией «раздел → компонент → группа параметров», как в консоли KSC:
  «хлебные крошки» с флагом достоверности, поиск, фильтр «только изменённые».
- **Трёхуровневое наследование** параметров с принудительной блокировкой («замочек»).
- **Списки-таблицы** (исключения, доверенные объекты, пакетные правила) через единый компонент,
  управляемый схемой из каталога.
- **Контроль устройств** — матрица режимов доступа, доверенные устройства по ID/маске,
  временный доступ, правила с расписанием.
- **Сравнение политик** — матрица «параметр × политика» с режимом «только отличия».
- **Управление изменениями** — заявки, ревизии, согласования, конечный автомат статусов.
- **Матрица покрытия атак** — 54 техники MITRE ATT&CK по 12 тактикам, честная 5-уровневая оценка:
  *блокируется / обнаруживается / частично / не включено / нет контроля*.
- **Подсказки по параметрам** — влияние на защиту и производительность по документации.
- Экспорт/импорт конфигурации политики, полностью русскоязычный интерфейс.

## Архитектура

Монорепозиторий (pnpm workspace), четыре слоя:

| Слой | Технологии | Назначение |
|------|-----------|-----------|
| Данные | PostgreSQL, Prisma ORM | Схема, миграции, наполнение |
| Предметный | TypeScript (чистые функции) | Наследование параметров, расчёт покрытия |
| Серверный | tRPC, Express | Типобезопасный API клиент↔сервер |
| Клиентский | React, Vite, Tailwind CSS | Интерфейс (офлайн, на русском) |

**Принцип «каталоги как данные».** Справочники параметров, событий, критериев, векторов атак и
навигации хранятся в YAML (`catalogs/`) и загружаются в БД. Добавление версии KESL или нового
списка — это правка данных, а не кода.

## Структура репозитория

```
apps/
  api/            tRPC-сервер (Express)
  web/            React-приложение (Vite)
packages/
  db/             Prisma-схема + сид (@ksc/db)
  domain/         чистая бизнес-логика: наследование, покрытие (@ksc/domain)
catalogs/         справочные данные в YAML (параметры, события, векторы атак, маппинги…)
infra/            Dockerfile.api, Dockerfile.web, nginx.conf
docs/             документация
build_catalogs.py генератор каталогов из исходной таблицы KSC (опционально)
dev.sh            запуск всего стека одной командой
docker-compose.yml
```

## Требования

- Node.js ≥ 20, pnpm ≥ 9
- Docker (для локальной PostgreSQL)

## Быстрый старт

```bash
pnpm install
cp .env.example .env
./dev.sh
```

`dev.sh` сам поднимает PostgreSQL в Docker, применяет схему и наполняет БД, затем запускает
API на `http://localhost:3001` и веб-интерфейс на `http://localhost:5173`.

## Ручной запуск

```bash
pnpm install
cp .env.example .env
docker compose up -d postgres

export DATABASE_URL="postgresql://ksc:ksc_dev_pw@localhost:5432/ksc_mgmt"
pnpm --filter @ksc/db exec prisma db push        # схема
pnpm --filter @ksc/db exec tsx src/seed.ts       # наполнение данными

pnpm dev                                          # api + web
```

> `ksc_dev_pw` — локальный dev-пароль PostgreSQL из `docker-compose.yml`, используется только
> для разработки. Реальных секретов в репозитории нет; рабочие значения задаются через `.env`.

## Матрица покрытия атак

Ключевая особенность — **честная** оценка защищённости. Покрытие каждой техники MITRE ATT&CK
считается по *действующей* политике и попадает в один из пяти статусов:

- **блокируется** — контроль реально останавливает технику;
- **обнаруживается** — техника видна в событиях, но не блокируется;
- **частично** — покрыты лишь отдельные варианты;
- **не включено** — контроль есть, но выключен в политике (лечится настройкой);
- **нет контроля** — техника вне зоны действия EPP, нужен другой рубеж (EDR, PAM, сегментация).

Оценка не завышается: техники, которые KESL закрыть не может, честно помечены как пробелы.

## Ограничения

- Аутентификация пользователей самой системы пока не реализована.
- Импортированные политики не привязаны к группам администрирования (в планах).
- Матрица оценивает конфигурацию политики, а не поведение защиты в реальном времени.
